**Author:** samacqua | **State:** MERGED | **BPB:** 1.195 | **Diff:** 1724 lines

TECHNIQUES:
- [EVAL] doc_isolated: Evaluate each document in isolation (reset context at BOS), no cross-doc token contamination. Delta: -0.011 BPB
- [EVAL] sliding_window_chunked: Chunk docs into 256-token strides within 1024-token context windows. Delta: -0.034 BPB (cumulative with doc isolation)
- [EVAL] lora_ttt: Per-document rank-8 LoRA adapters on `lm_head`, `c_q`, `c_v` across all layers. Score chunk → train on chunk loss → advance (no leakage). 1 Adam step per chunk, lr=0.01. Delta: -0.037 BPB total (LoRA itself only adds ~-0.003 over sliding window)

KEY_INNOVATIONS:
- Batched LoRA TTT (bsz=64, sort by length) enables per-doc independent adaptation without sequential overhead — ~5x faster than full-param TTT while maintaining no cross-doc leakage
- LoRA targets only Q/V projections + lm_head (not K), keeping adapter parameter count small while hitting the highest-impact projections

WE_DONT_HAVE:
- Doc-isolated eval boundary handling (explicit BOS detection to reset context per document — unclear if our sliding window already does this or still processes a flat token stream)
- Batched LoRA TTT specifically (we have full-param SGD TTT with prefix/suffix split, which is a different causal structure)

WE_SHOULD_STEAL:

- **LoRA TTT over full-param TTT** (medium): Our full-param SGD likely gets slightly better BPB per document, but LoRA's 64x batching means we can run far more TTT steps within the eval time budget. Net BPB may favor LoRA if we're currently bottlenecked on TTT wall-clock time.
